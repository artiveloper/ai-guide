# ai-guide — 아키텍처 리뷰 하네스

여러 프로젝트가 공통으로 재사용하는 **아키텍처/품질 리뷰 지식**을 에이전트·스킬로 관리하는 저장소입니다. 각 에이전트는 독립적으로 호출되며(에이전트 간 통신 없음), 필요할 때만 개별적으로 사용합니다.

## 설계 철학

- **스택 무관 원칙만** 담는다. 언어·프레임워크에 종속된 규칙(예: Next.js App Router, Prisma, Supabase 인증)은 여기 넣지 않고 별도 기술 스킬로 분리한다.
- **에이전트는 얇게, 원칙은 스킬에** 둔다. 에이전트는 역할·판단 태도만 정의하고, 실제 판단 기준은 짝이 되는 스킬에서 로드한다.
- **오케스트레이션은 프로젝트 몫이다.** 여러 에이전트를 순서대로 엮는 것과 프로젝트 특이사항은 각 프로젝트의 `.claude/`에 복사해 확장한다. 이 저장소의 원본은 항상 범용성을 유지한다.

## 구성

도메인별로 `agents/`(에이전트 정의)와 `skills/`(판단 기준)를 짝지어 둔다.

### `common/` — 횡단(시스템) 설계

| 에이전트 | 스킬 | 담당 |
|----------|------|------|
| `architect` | `system-architecture` | 요구사항(FR/NFR) 구조화, 규모별 기술 스택 선정 규율, KISS·확장 지점, 도메인 간 경계 조율 |

### `back/` — 백엔드

| 에이전트 | 스킬 | 담당 |
|----------|------|------|
| `backend-architect` | `backend-architecture` | 계층 분리, API 계약·응답 봉투, 입력 검증, 트랜잭션 경계, 인증/인가, 보안 기본, 에러 처리 |
| `dba-advisor` | `db-architecture` | 정규화, 인덱스 전략, 마이그레이션 안전성, N+1/락 이슈 |
| `qa-backend` | `qa-backend-strategy` | API 계약 검증, 통합 테스트 우선순위, 동시성·회귀 방지 |

> 기술 스킬: `supabase-guide`, `python-guide`

### `front/` — 프론트엔드

| 에이전트 | 스킬 | 담당 |
|----------|------|------|
| `frontend-architect` | `frontend-architecture` | 상태 분류, 컴포넌트 경계, 데이터 fetching, 낙관적 업데이트 롤백, 에러 경계, 폼 검증 이중화 |
| `qa-frontend` | `qa-frontend-strategy` | 행동 기반 컴포넌트 테스트, E2E 우선순위, 안정적 셀렉터 |
| `design-system` | `design-system` | 디자인 토큰, 컴포넌트 재사용, 반응형/모바일 퍼스트, 접근성, 로딩·빈·에러 상태 |

> 기술 스킬: `nextjs-guide`, `react-query-guide`, `shadcn-ui`

## 사용 방법

백엔드/프론트엔드/QA/디자인시스템/DB 아키텍처의 설계나 리뷰가 필요할 때 해당 에이전트를 개별 호출한다. 에이전트는 짝이 되는 스킬을 먼저 로드해 판단 기준을 확인한 뒤, 구조적 결함을 지적하고 우선순위와 트레이드오프를 함께 제시한다.

## 확장 (프로젝트 적용)

프로젝트별 특수 규칙이 필요하면:

1. 해당 에이전트/스킬 파일을 프로젝트의 `.claude/agents/`·`.claude/skills/`에 복사한다.
2. 파일 하단에 `## 프로젝트 특이사항` 섹션을 추가해 확장한다.
3. 여러 에이전트를 묶어 실행하는 오케스트레이션은 그 프로젝트의 하네스에서 구성한다.

원본(이 저장소)은 스택 무관 범용 원칙만 유지한다.
