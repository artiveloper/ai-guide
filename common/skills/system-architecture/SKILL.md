# System Architecture

스택 불문 상위(시스템) 설계 판단 기준. 도메인별 세부(백엔드 계층·DB 스키마·프론트 상태 등)는 `backend-architecture`·`db-architecture`·`frontend-architecture` 스킬이 담당하고, 이 스킬은 그 위에서 **요구사항 → 구조 → 스택**을 잇는 판단만 다룬다. 특정 스택 권장(예: Next.js, Prisma)이나 산출물 문서 포맷은 이 스킬에 넣지 않고 프로젝트/기술 스킬에 위임한다.

## 요구사항 구조화

- 기능 요구사항(FR)과 비기능 요구사항(NFR)을 분리해 명시한다 — 모든 설계 판단은 이 둘에서 출발한다.
- FR에는 우선순위(P0/P1/P2 등)를 붙여 MVP 경계를 분명히 한다. 무엇을 지금 만들지 않을지가 무엇을 만들지만큼 중요하다.
- NFR(성능·가용성·보안·확장성·규정)은 측정 가능한 수치·기준으로 표현한다("빠르게"가 아니라 "p95 200ms", "동시 접속 1k"). 측정 불가능한 NFR은 설계 판단에 쓸 수 없다.
- 요구사항이 모호하면 침묵의 가정을 두지 말고 가정을 명시적으로 문서화한 뒤 진행한다.

## 아키텍처 판단 원칙

- **KISS**: 요구사항을 충족하는 가장 단순한 구조를 택한다. 현재 요구에 없는 복잡성(마이크로서비스, 이벤트 소싱 등)을 미리 넣지 않는다.
- **확장 지점 명시**: 지금은 단순하게 가되, 규모가 커질 때 어디를 어떻게 바꿀지(확장 지점)를 문서에 남긴다 — 만들지는 않되 자리는 비워둔다.
- 트레이드오프가 있는 결정(동기 vs 비동기, 모놀리식 vs 분리, SQL vs NoSQL)은 선택지·근거·비용을 함께 제시하고 단정하지 않는다.
- 설계는 팀이 곧바로 구현에 착수할 수 있을 만큼 구체적이어야 한다 — 계층 경계·소유권·인터페이스가 모호하면 설계가 덜 된 것이다.

## 기술 스택 선정

- 스택은 유행이 아니라 요구사항과 규모(MVP/성장/대규모)에 맞춰 선정하고, 각 선택에 근거와 트레이드오프를 명시한다.
- 팀의 기존 역량·운영 부담·생태계 성숙도를 요구사항과 함께 고려한다 — 가장 강력한 기술이 아니라 가장 적합한 기술을 고른다.
- 되돌리기 어려운 결정(DB 종류, 핵심 프레임워크, 인증 모델)과 나중에 바꾸기 쉬운 결정을 구분해, 전자에 판단 비용을 집중한다.
- 구체적 스택 권장은 프로젝트 스택 스킬에 위임한다 — 이 스킬은 "규모·요구에 맞춰 근거를 대고 고른다"는 선정 규율만 다룬다.

## 보안·횡단 관심사를 설계 단계에 포함

- 인증/인가 모델, 입력 신뢰 경계, 비밀·환경변수 관리, CORS 등 횡단 관심사는 구현이 아니라 설계 단계에서 결정한다(세부는 `backend-architecture` 스킬).
- 데이터 경계(어떤 데이터가 어디에 저장·전송되는지)와 민감 데이터 취급을 초기에 정의한다.

## 도메인 설계 위임

- API 계약·계층 분리·트랜잭션 경계 → `backend-architecture`
- 스키마·정규화·인덱스·마이그레이션 → `db-architecture`
- 상태 분류·컴포넌트 경계·데이터 fetching → `frontend-architecture`
- 이 스킬은 위 산출물이 요구사항(FR/NFR)과 정합하는지, 도메인 간 경계(누가 무엇을 소유하는지)가 충돌 없는지를 상위에서 점검한다.

## 리뷰 시 체크 우선순위

1. FR/NFR이 구조화되어 있고 NFR이 측정 가능한가
2. 구조가 요구에 비해 과설계/과소설계가 아닌가 (KISS, 확장 지점 명시)
3. 되돌리기 어려운 결정에 근거·트레이드오프가 명시됐는가
4. 보안·데이터 경계가 설계 단계에서 다뤄졌는가
5. 도메인 간 소유권/경계가 충돌 없이 정의됐는가
