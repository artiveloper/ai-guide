# Backend Architecture

스택(Python/Node/Go 등)에 무관하게 적용되는 백엔드 구조 원칙. 언어별 세부 컨벤션은 별도 스킬(예: python-clean-code)을 참고하고, 이 스킬은 구조적 판단 기준만 다룬다. 특정 프레임워크·기술(예: Supabase 인증, Pterodactyl 연동)에 종속된 규칙은 이 스킬에 넣지 않고 별도 기술 스킬로 분리해 하네스가 매칭한다.

## 계층 분리

- **Controller/Handler**: 요청 파싱, 응답 포맷팅만 담당. 비즈니스 로직을 넣지 않는다.
- **Service**: 비즈니스 로직의 단일 소유자. 여러 리소스에 걸친 트랜잭션 경계도 여기서 관리한다.
- **Repository/DAO**: 데이터 접근만 담당. 여기에 비즈니스 규칙이 새어 들어가면(예: 조건부 필터링 로직) 서비스 계층으로 옮긴다.

판단 기준: 상위 계층은 하위 계층의 구현을 몰라야 한다(의존성은 항상 안쪽 → 바깥쪽이 아니라 controller → service → repository 한 방향). 역방향 의존(repository가 service를 호출)이 보이면 구조적 결함이다.

## API 계약 설계

- 응답 스키마는 프론트엔드와 사전 합의하고, 버전 변경 시 하위 호환을 깨는지 명시적으로 판단한다.
- 에러 응답 포맷은 프로젝트 전체에서 일관되어야 한다(코드, 메시지, 상세 필드의 위치가 엔드포인트마다 다르면 결함).
- 멱등성이 필요한 엔드포인트(결제, 생성 등)는 명시적으로 표시하고 검증한다.
- 성공/에러를 구분 가능한 일관된 응답 봉투(envelope)를 프로젝트 전체에서 하나로 정한다. 형태 예시(언어 무관):
  - 성공: `{ success: true, data, meta }` — 목록이면 `meta`에 `page`/`total` 등 페이지네이션 정보를 담는다.
  - 에러: `{ success: false, error: { code, message, details } }` — `code`는 기계 판독용, `message`는 사람 판독용으로 역할을 분리한다.
- 목록 응답은 페이지네이션 메타와 결과 상한을 강제한다(무제한 조회 금지). *성능 관점의 페이지네이션은 `db-architecture` 스킬 참조 — 여기서는 계약 일관성 관점만 다룬다.*

## 입력 검증 (경계면)

- 모든 외부 입력(body/query/params/header)은 도메인 로직 진입 전, controller/handler 경계에서 스키마로 검증한다. 신뢰 경계 안쪽에는 검증된 타입만 흐르게 한다.
- 검증 실패는 즉시 표준 에러 봉투로 반환하고(예: 400), 어떤 필드가 왜 실패했는지 `details`로 제공한다.
- 검증 라이브러리는 스택별 선택사항(예: 스키마 검증 라이브러리) — 원칙은 "경계에서 스키마 검증, 안쪽은 신뢰".

## 트랜잭션 경계

- 트랜잭션은 서비스 계층에서 시작하고 끝나야 한다. Repository가 자체적으로 커밋하면 여러 리소스에 걸친 원자성이 깨진다.
- 외부 API 호출(이메일 발송, 결제 게이트웨이 등)은 트랜잭션 안에 넣지 않는다 — 실패 시 롤백 불가능하거나 트랜잭션이 불필요하게 길어진다.
- 긴 트랜잭션은 락 경합을 유발한다. 읽기 전용 작업은 트랜잭션 밖으로 뺄 수 있는지 검토한다.

## 에러 처리 전략

- 예외를 계층마다 다르게 잡지 말고, 하나의 중앙 에러 핸들러(미들웨어)에서 HTTP 상태 코드로 변환한다.
- 비즈니스 예외(예: "재고 부족")와 시스템 예외(DB 연결 실패)를 구분해서 다른 로그 레벨/알림 정책을 적용한다.
- 에러를 삼키지 않는다 — catch 후 아무 처리 없이 넘어가는 코드는 결함으로 간주한다.

## 인증 / 인가

- 인증(누구인가)과 인가(무엇을 할 수 있는가)를 분리한다. 인가는 서비스/미들웨어 계층에서 일관되게 수행한다.
- 신원·권한은 신뢰 가능한 서버 측 소스에서만 취한다 — 클라이언트가 보낸 값(role 등)을 그대로 신뢰하지 않는다.
- 소유권 검증: 리소스 접근 시 요청자 소유 여부를 확인하고, 불일치 시 리소스 존재 여부를 노출하지 않고 차단한다(403/404).
- RBAC 등 역할 기반 접근 제어는 한 곳에서 정의하고 엔드포인트마다 재정의하지 않는다.

## 보안 기본 원칙

- 인젝션 방지: 쿼리는 파라미터 바인딩/ORM으로 작성하고, 사용자 입력을 문자열 연결로 쿼리에 넣지 않는다.
- 시크릿 관리: 비밀키·토큰은 환경변수로 관리하고 서버 전용으로 유지한다(클라이언트 번들·로그 노출 금지). 환경변수는 런타임에 존재·형식을 검증한다.
- 민감 데이터 비노출: 응답에는 필요한 필드만 반환하고(내부 ID·메모 등 제외), 내부 예외 메시지를 클라이언트에 그대로 노출하지 않는다.
- 로그 마스킹: 비밀키·토큰·PII는 평문 로깅 금지.
- 크리덴셜은 단방향 해싱(솔트 포함)으로 저장한다. CORS·레이트리밋 등 경계 보호를 명시적으로 설정한다.

## 리뷰 시 체크 우선순위

1. 계층 간 의존성 방향이 지켜지는가
2. 트랜잭션 경계가 명확한가 (특히 외부 호출이 섞여 있는지)
3. 에러 처리가 중앙화되어 있는가
4. API 응답 계약이 일관적인가 (응답 봉투 통일)
5. 외부 입력이 경계에서 스키마 검증되는가
6. 인가·소유권 검증이 일관되게 적용되는가
7. 시크릿/민감 데이터 노출·인젝션 방지가 지켜지는가
