# QA Backend Strategy

특정 기술·프로젝트(예: Supabase RLS 검증, Pterodactyl 전원 제어 시나리오)에 종속된 검증 항목은 이 스킬에 넣지 않고 별도 기술 스킬로 분리해 하네스가 매칭한다. 이 스킬은 스택 불문 테스트 전략 판단 기준만 다룬다.

## 테스트 피라미드 판단 기준

- **단위 테스트**: 순수 로직(가격 계산, 검증 규칙 등)에 집중. DB/네트워크 의존이 있으면 단위 테스트가 아니다.
- **통합 테스트**: 실제 DB(테스트용 인스턴스)를 사용해 repository/service 계층의 실제 쿼리 동작을 검증. 여기서 mock DB를 쓰면 실제 SQL 오류나 제약조건 위반을 못 잡는다.
- **E2E**: API 엔드포인트 전체 흐름 중 핵심 경로(결제, 인증)만 — 모든 엔드포인트를 E2E로 덮으려 하지 않는다.

## Mock 기준

- 외부 서드파티 API(결제 게이트웨이, 이메일 발송)는 mock 처리한다 — 테스트가 외부 서비스 가용성에 의존하면 안 된다.
- 자체 DB는 mock하지 않는다 — 실제 쿼리 문법 오류, 제약조건 위반은 mock으로 잡을 수 없다. 대신 테스트 전용 DB/트랜잭션 롤백 전략을 쓴다.
- 시간(현재 시각)에 의존하는 로직은 반드시 주입 가능하게 만들고 테스트에서 고정한다.

## 반드시 검증해야 할 경계면

1. **API 계약**: 요청 스키마 검증 실패 시 정확한 에러 응답이 오는가
2. **트랜잭션 실패 케이스**: 중간에 실패했을 때 롤백이 제대로 되는가 (부분 커밋 여부)
3. **동시성**: 같은 리소스에 대한 동시 요청이 레이스 컨디션을 일으키지 않는가 (재고 차감, 잔액 갱신 등)
4. **권한 경계**: 다른 사용자의 리소스에 접근 시도 시 차단되는가

## 흔한 갭 패턴

- Happy path만 테스트하고 실패 경로(네트워크 타임아웃, 잘못된 입력, 권한 없음)가 비어있는 경우가 가장 흔하다.
- 마이그레이션에 대한 테스트가 없어 스키마 변경이 기존 쿼리를 깨뜨리는 걸 못 잡는 경우.

## 리뷰 시 체크 우선순위

1. 실패 경로(에러/권한/동시성) 테스트 존재 여부
2. Mock 사용이 적절한 경계에서 이루어지는가
3. 테스트 피라미드 비율이 뒤집혀 있지 않은가(E2E 과다, 단위 부족)
