# DB Architecture

DBMS(PostgreSQL, MySQL 등)에 무관하게 적용되는 원칙. 특정 DBMS 확장 기능이나 특정 기술(예: Supabase RLS)에 종속된 규칙은 이 스킬에 넣지 않고 별도 기술 스킬로 분리해 하네스가 매칭한다.

## 정규화 판단

- 정규화는 무조건 높일수록 좋은 게 아니다 — 읽기가 압도적으로 많은 테이블은 의도적 비정규화(중복 저장)로 조인을 줄이는 것이 합리적일 수 있다.
- 비정규화를 선택했다면 데이터 일관성을 어떻게 유지할지(트리거, 애플리케이션 레벨 동기화)를 반드시 함께 설계한다 — 동기화 전략 없는 비정규화는 결함이다.

## 인덱스 전략

- WHERE, JOIN, ORDER BY에 자주 쓰이는 컬럼에 인덱스가 있는지 확인한다.
- 인덱스는 쓰기 성능을 희생한다 — 쓰기가 빈번한 테이블에 인덱스를 과도하게 추가하지 않는다.
- 복합 인덱스는 컬럼 순서가 중요하다 — 쿼리 패턴과 맞지 않는 순서면 인덱스가 활용되지 않는다.

## 마이그레이션 안전성

- 운영 중인 대형 테이블에 컬럼 추가/변경 시 테이블 전체 락이 걸리는지 확인한다 (DBMS/버전에 따라 다름) — 필요하면 온라인 마이그레이션 기법을 검토한다.
- 컬럼 삭제/이름 변경은 즉시 실행하지 않고 단계적으로(사용 중단 확인 → 삭제) 진행한다 — 배포된 애플리케이션 코드가 아직 참조 중일 수 있다.
- 모든 마이그레이션은 롤백 경로가 있어야 한다.

## 흔한 성능 함정

- **N+1 쿼리**: 리스트를 순회하며 각 항목마다 별도 쿼리를 날리는 패턴 — eager loading이나 배치 쿼리로 전환한다.
- **풀 테이블 스캔**: 인덱스 없이 대형 테이블을 필터링하는 쿼리.
- **긴 트랜잭션 + 락**: 트랜잭션 안에서 외부 API를 호출하거나 불필요하게 오래 유지하면 락 경합을 유발한다.

## 리뷰 시 체크 우선순위

1. 마이그레이션이 운영 환경에 안전한가 (락, 롤백 가능성)
2. N+1/풀스캔 패턴이 있는가
3. 정규화 수준이 읽기/쓰기 패턴과 맞는가
4. 인덱스가 실제 쿼리 패턴을 반영하는가
