# Frontend Architecture

프레임워크(Next.js/Vue 등)에 무관하게 적용되는 프론트엔드 구조 원칙. 특정 프레임워크·라이브러리(예: Next.js App Router, React Query, nuqs)에 종속된 규칙은 이 스킬에 넣지 않고 별도 기술 스킬로 분리해 하네스가 매칭한다.

## 상태 분류 — 가장 먼저 판단할 것

모든 상태 관리 문제는 아래 4가지 중 어디에 속하는지 먼저 분류하는 데서 시작한다:

1. **서버 상태**: API로부터 온 데이터(사용자 목록, 게시글 등). 캐싱/재검증 로직이 필요 — 전역 스토어에 수동으로 넣지 않는다.
2. **URL 상태**: 필터, 페이지네이션, 탭 선택 등 새로고침 시에도 유지되어야 하는 상태 — URL 쿼리 파라미터로 관리한다.
3. **폼 상태**: 입력 중인 값 — 로컬 컴포넌트 상태로 충분하다.
4. **전역 UI 상태**: 다크모드, 사이드바 열림/닫힘 등 진짜 앱 전역 상태만 전역 스토어에 둔다.

가장 흔한 결함: 서버 상태를 전역 스토어에 수동 캐싱하면서 재검증/무효화 로직을 직접 구현 — 캐시 무효화 버그가 여기서 발생한다.

## 컴포넌트 경계

- **Presentational vs Container** 구분: 데이터를 가져오는 로직과 렌더링 로직을 분리한다. 그래야 같은 UI를 다른 데이터 소스로 재사용할 수 있다.
- 하나의 컴포넌트가 너무 많은 책임(데이터 fetching + 상태 관리 + 복잡한 렌더링 분기)을 가지면 분리 신호다. 파일이 비대해지는 것 자체도 책임 과다 신호로 본다.
- Props가 과도하게 많아지면(대략 5개 초과가 신호) 관련 값을 객체로 묶거나 컴포지션으로 분해한다.
- Props drilling이 3단계 이상 깊어지면 컴포지션(children prop) 또는 컨텍스트를 고려한다.
- 여러 곳에서 반복되는 비시각적 로직(데이터 변환, 부수효과 등)은 컴포넌트에서 분리해 재사용 단위(훅/컴포저블 등)로 추출한다 — 렌더링과 로직을 뒤섞지 않는다.

## 데이터 Fetching 패턴

- 컴포넌트 마운트 시 직접 fetch하는 패턴(useEffect + fetch)은 경쟁 조건, 캐싱 부재 문제가 있다 — 전용 라이브러리(react-query, SWR 등)나 프레임워크 내장 fetching을 우선한다.
- 워터폴(하위 컴포넌트가 순차적으로 데이터를 요청) 발생 여부를 확인한다 — 병렬화 가능하면 상위에서 한 번에 요청한다.
- 데이터 변경(mutation)은 로딩·에러·성공 상태를 각각 처리한다. 낙관적 업데이트(옵티미스틱 UI)를 쓰면 **실패 시 이전 상태로 되돌리는 롤백 경로를 반드시 함께 설계**하고, 서버 확정 응답으로 최종 상태를 재조정한다 — 롤백 없는 낙관적 업데이트는 결함이다.

## 에러 경계 · 폼 검증

- **에러 경계(Error Boundary)**: 페이지/기능 단위로 렌더 에러 경계를 두어, 한 컴포넌트의 실패가 화면 전체를 무너뜨리지 않도록 격리한다. *에러 "상태" UI 표현(빈/에러 구분)은 design-system 스킬 소관 — 여기서는 실패 격리라는 구조 관점만 다룬다.*
- **폼 검증 이중화**: 검증은 클라이언트·서버 양쪽에 둔다. 클라이언트 검증은 즉각 피드백(UX)용일 뿐 신뢰 경계가 아니다 — 서버 검증을 최종 관문으로 삼고 클라이언트 검증만 신뢰하지 않는다(백엔드 입력 검증과 짝을 이룬다).

## 폴더 구조 원칙

- **기능별(feature-based)** 구조를 기본으로 한다 — 타입별(components/, hooks/, utils/) 구조는 프로젝트가 커지면 기능 하나 수정에 여러 폴더를 오가야 해서 확장성이 떨어진다.
- 공통 UI 프리미티브(버튼, 인풋 등)만 별도 `shared/ui` 같은 위치에 둔다.

## 리뷰 시 체크 우선순위

1. 서버 상태가 올바른 방식으로 관리되는가 (수동 캐싱 남용 여부)
2. 컴포넌트가 단일 책임을 지키는가 (props 과다·비대한 파일 여부)
3. 데이터 fetching에 워터폴/경쟁 조건이 없는가
4. 폴더 구조가 기능 단위로 확장 가능한가
5. 뮤테이션/낙관적 업데이트에 로딩·에러·롤백 처리가 있는가
6. 페이지/기능 단위 에러 경계가 있는가
7. 폼 검증이 클라이언트·서버 양쪽에 있는가 (클라이언트 검증만 신뢰하지 않는가)
