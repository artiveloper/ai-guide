# Design System

특정 UI 라이브러리·프레임워크(예: shadcn/@workspace/ui, Tailwind)에 종속된 규칙은 이 스킬에 넣지 않고 별도 기술 스킬로 분리해 하네스가 매칭한다. 이 스킬은 라이브러리 불문 디자인 시스템 판단 기준만 다룬다.

## 디자인 토큰

- 색상, 간격, 타이포그래피, 그림자 등은 하드코딩된 값(`#3B82F6`, `16px`)이 아니라 이름 있는 토큰(`color-primary`, `space-md`)으로 참조되어야 한다.
- 토큰이 있어도 컴포넌트가 임의로 override하면 시스템이 무력화된다 — override 빈도가 높은 컴포넌트는 토큰 자체가 부족하다는 신호다.
- 다크모드/테마 전환이 필요하면 토큰은 값이 아니라 semantic 이름(예: `surface-primary`)으로 정의해야 테마별 값만 바꿔 대응할 수 있다.

## 컴포넌트 재사용성 판단

- 같은 시각적 목적(버튼, 카드, 뱃지)의 컴포넌트가 여러 곳에서 독립적으로 구현되어 있으면 중복 — 하나로 통합하고 variant/prop으로 차이를 표현한다.
- 재사용 가능한 컴포넌트는 스타일과 동작을 분리한다 — 특정 페이지 전용 로직이 공용 컴포넌트에 섞이면 재사용성이 깨진다.

## 반응형 / 모바일 퍼스트

- 모든 UI는 **모바일 화면을 기준으로 먼저 설계**하고 큰 화면으로 확장한다(mobile-first). 데스크탑을 먼저 설계하고 축소하지 않는다.
- 인터랙티브 요소(버튼·링크)의 **터치 타겟은 최소 44px** 확보한다 — 손가락 터치 대응(접근성과도 직결).
- 좁은 화면에서 넓은 테이블은 **가로 스크롤 또는 카드 목록**으로 대체한다. 콘텐츠를 잘리게 하거나 모바일 스크롤 자체를 막지 않는다.

## 로딩·빈·에러 상태

- 로딩 상태는 최종 콘텐츠의 **레이아웃 형태를 유지하는 스켈레톤**으로 표현한다. "불러오는 중..." 같은 단순 텍스트로 대체하지 않는다 — 레이아웃 점프(CLS)와 인지 부하를 줄인다.
- 스켈레톤은 화면마다 새로 만들지 말고 **공용 컴포넌트**(테이블·카드·헤더)로 재사용한다.
- **빈 목록**과 **에러 상태**는 각각 명시적으로 처리한다(데이터 없음 안내 / 실패 안내). 로딩·빈·에러 셋을 구분해 서로 다른 UI로 보여준다.

## 상태 표현의 일관성

- 엔티티의 상태값(실행 중/중지/오류 등)은 **라벨과 색상(variant)의 매핑을 한 곳에서 중앙 정의**해 재사용한다. 화면마다 색·문구를 즉흥적으로 정하지 않는다.
- **색상만으로 상태를 구분하지 않는다** — 라벨(텍스트)을 함께 제공해 색각 이상 사용자도 구분할 수 있게 한다(접근성).

## 접근성 — 필수 체크 항목

1. 색상 대비가 WCAG AA 기준(일반 텍스트 4.5:1)을 만족하는가
2. 인터랙티브 요소가 키보드만으로 접근 가능한가(포커스 순서, 포커스 표시)
3. 의미 있는 마크업(버튼은 `<button>`, 제목은 `<h1>~<h6>`)을 쓰는가 — `<div onClick>`으로 대체된 버튼은 스크린리더에서 인식 안 됨
4. 이미지/아이콘에 대체 텍스트가 있는가

## 리뷰 시 체크 우선순위

1. 하드코딩된 스타일 값 존재 여부 (토큰화 누락)
2. 동일 목적 컴포넌트의 중복 구현 여부
3. 접근성 필수 항목 위반 여부
4. 모바일 퍼스트·터치 타겟(44px) 위반 여부
5. 로딩·빈·에러 상태 처리 누락 여부 (스켈레톤 대신 단순 텍스트, 빈/에러 미구분)
6. 상태 표현이 화면마다 제각각인지 (중앙 매핑 부재, 색상 단독 의존)
