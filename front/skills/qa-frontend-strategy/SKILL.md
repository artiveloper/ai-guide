# QA Frontend Strategy

특정 프레임워크·라이브러리(예: React Query 패턴 검증, Next.js 상태 전이 UI)에 종속된 검증 항목은 이 스킬에 넣지 않고 별도 기술 스킬로 분리해 하네스가 매칭한다. 이 스킬은 프레임워크 불문 테스트 전략 판단 기준만 다룬다.

## 기본 원칙 — 구현이 아니라 행동을 테스트

- 컴포넌트의 내부 state나 함수 호출 여부를 검증하는 테스트는 리팩터링마다 깨진다 — "사용자가 버튼을 누르면 화면에 X가 보인다" 같은 관찰 가능한 행동을 검증한다.
- 스냅샷 테스트는 UI가 자주 바뀌는 컴포넌트에 남용하면 실패해도 아무도 diff를 제대로 검토하지 않는 "가짜 안전망"이 된다 — 구조가 안정적인 컴포넌트에만 제한적으로 쓴다.
- 셀렉터는 접근성 역할·라벨·텍스트 쿼리를 우선한다. 불가피할 때만 안정적인 test id를 쓰고, 구현 세부(클래스명·DOM 구조)에 의존하는 셀렉터는 피한다 — 리팩터링마다 깨진다. (안정적 셀렉터 확보를 위해 대상 요소에 test id를 부여하는 것은 구현 측과 합의한다.)

## 테스트 레벨별 역할

- **단위 테스트**: 순수 로직(포맷터, 계산 함수)
- **컴포넌트 테스트**: 사용자 상호작용(클릭, 입력) → 화면 변화. API는 mock.
- **E2E**: 여러 페이지/컴포넌트를 가로지르는 핵심 사용자 흐름만 (로그인, 결제, 핵심 CRUD). 모든 페이지를 E2E로 덮지 않는다 — 느리고 깨지기 쉽다.

## E2E 우선순위 선정 기준

1. 실패 시 비즈니스 임팩트가 큰 흐름(결제, 가입)
2. 여러 시스템(프론트+백엔드+외부 API)이 연결되는 흐름 — 단위/컴포넌트 테스트로는 못 잡는 통합 문제
3. 자주 회귀가 발생했던 흐름(과거 버그 이력 기준)

## 흔한 갭 패턴

- 로딩/에러 상태 UI가 테스트되지 않는 경우(항상 성공 응답만 가정)
- 폼 유효성 검증의 실패 케이스가 비어있는 경우
- 접근성(키보드만으로 조작 가능한지)이 테스트에서 아예 빠져있는 경우 — design-system 스킬과 겹치는 영역이니 교차 확인한다.

## 리뷰 시 체크 우선순위

1. 구현 세부사항이 아니라 사용자 행동을 검증하는가
2. 로딩/에러 상태가 테스트되는가
3. E2E가 핵심 흐름에 집중되어 있는가 (과다/부족 여부)
